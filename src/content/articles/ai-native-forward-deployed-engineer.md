AI-native engineering is not only about calling a model API. The useful work is turning a real workflow into a system that can be deployed, observed, reviewed, and improved.

My strongest fit is the forward deployed side of that work: understand the user workflow, identify where AI or automation belongs, and build the backend path around it. That usually means service boundaries, event queues, storage, validation, retries, security checks, and a rollout plan that a delivery team can operate.

The production foundation matters. I have built Golang services, gRPC layers, Kafka workflows, Google Pub/Sub and Azure Service Bus integrations, Redis-backed performance improvements, Kubernetes delivery, Terraform automation, and CI/CD pipelines with security scanning.

For an AI-native product team, that background helps connect product ideas to reliable implementation. The first version should be fast enough to learn from users, but structured enough that the team can keep shipping after the first deployment.
